# 📜 Project Nomad: 游戏设计核心规范 (Game Design Document)

> 本文档基于已确认的核心设定进行整合：**写实奇幻**、**灵魂投射战斗**、**参数化格点逻辑**、**游牧部族**。  
> 目的：作为 Demo 的设计底座与实现约束。

---

## 1. 世界观与主题 (Worldbuilding & Theme)
* **核心冲突：** 奇幻背景下的写实生存。世界充满自然/灵魂能量，但传统“魔法师”极度罕见。
* **社会形态：** 游牧部族（数十至百人规模）。主人公作为部族成员，在大地图上随部族动态移动。
* **生物设定：** 宝可梦是自然的组成部分，遵循生老病死。它们因更擅长运用自然能量而拥有“技能”。
* **美术风格：** 追求写实质感（Realistic Texture），非传统卡通化设定。

## 2. 角色与共鸣机制 (Character & Resonance)
* **灵魂共鸣（收服）：** 并非捕捉，而是通过灵魂调频达成共鸣的“奇幻养宠”过程。
* **属性枷锁：** 主人公的属性特质直接决定了可共鸣宝可梦的风格。属性不合将导致加成缺失甚至无法共鸣。
* **灵魂投射（战斗）：**
  * 战斗时主人公将意识投射进宝可梦体内进行 1v1 对战。
  * **非暴力原则：** 战斗不会直接攻击对战者本人，因为意识在宝可梦体内，且本体死亡会导致对方更彻底的追杀。

## 3. 战斗系统：3D 参数化格点 (Tactical Grid System)
所有战斗逻辑必须在高度参数化的物理空间内结算。

### 3.1 空间定义 (Spatial System)
* **平面：** 六边形格点（Hex Grid）。
* **垂直：** 块高度（Block），1 Block = 1.0m。
* **体积占用（Occupancy）：**
  * 人类与小体型宝可梦：占用 $1\\text{ Hex} \\times 2\\text{ Blocks}$。
  * 所有生物的体积与技能范围均由 Hex 与 Block 的组合定义。

### 3.2 时间轴：30 拍系统 (Tick-Based System)
* **同步回合制：** 双方同时下达指令并同步执行。
* **原子化拍数：** 每回合固定为 **30 拍 (Ticks)**。
* **结算精度：** 技能与行为的结算固定到具体拍数（如第 $n/30$ 拍发生判定）。

## 4. 技术架构与表现 (Technical Architecture)
* **开发环境：** Godot 4.x + C#。
* **视角表现：** 2.5D/等轴测视角（Isometric），类似《星际争霸》或《博德之门》的轴侧表现。
* **模拟机制：** 采用 2D 模拟 3D 视角，通过 Godot 的 Y-Sort 和参数化格子处理深度排序与物理阻挡。

## 5. 交互与限制 (Interaction & Constraints)
* **探索限制：** 主人公一次只能带几只宝可梦离开部族进行探索。
* **动态原点：** 部族在大地图上缓慢移动，作为玩家探索的动态补给与存储中心。
* **行为可控性：** 所有的动画表现必须服务于参数化结算。虽然动画是连续的，但逻辑判定必须严格对齐 Tick。

---

## 🛠 开发优先级 (Technical Roadmap for Demo)

### 第一阶段：C# 逻辑底座
1. **Hex-Block 坐标系：** 实现 Cube Coordinates 到 Godot 世界坐标的映射。
2. **Tick 调度器：** 开发 `BattleClock` 类，驱动 0-29 拍的逻辑循环。
3. **体积判定：** 实现不同 Size 单位的静态占用逻辑。

### 第二阶段：同步执行逻辑
1. **并行解析：** 处理两个单位在同一拍移动或释放技能的判定冲突。
2. **技能模版：** 基于 Hex 集合与 Block 高度定义第一个参数化 AOE 技能。

### 第三阶段：轴侧视觉集成
1. **TileMap 配置：** 设置 Isometric Hex 模式。
2. **Y-Sort 验证：** 确保大体型生物移动时的正确遮挡。
